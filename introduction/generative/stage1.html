邊緣端寫入資料庫

無人機 / Jetson 完成推論後，必須將結果送上後端持久化，常見模式：

  • HTTPS POST：每次計數完成呼叫後端 REST API，適合低頻寫入。
  • MQTT：輕量級訊息協定，適合大量感測器同時上傳。
  • 批次上傳：無網路時暫存於本地 SQLite，連線恢復後一次同步。
  • 資料表設計：建議按時序欄位建立 index，並儲存原始照片連結而非 binary。

排程器定時觸發

後端不會每秒呼叫 LLM（成本太高），而是定期彙整數據：

  • APScheduler：Python 內嵌排程，適合小型專案。
  • Cron：Linux 原生排程，搭配 Docker 部署最穩定。
  • Apache Airflow：企業級資料管線，支援 DAG、重試、依賴管理。
  • 觸發時機：每小時短摘要、每日完整報告、異常即時觸發三層策略。

SQL 撈取歷史數據

查詢設計重點：

  • 時間窗：依報告類型選擇（即時 1h、日報 24h、週報 7d）。
  • 多表 JOIN：影像分析表 + 感測器表 + 圈舍配置表組合成完整視角。
  • 聚合運算：使用 AVG()、MAX()、PERCENTILE_CONT() 計算統計指標。
  • 效能：對時序欄位 ts 建立 B-tree 或 BRIN index，避免全表掃描。

整理為 JSON 結構

為什麼要轉成 JSON？因為 LLM 對結構化文字理解能力最佳：

  • 欄位語意化：用人類可讀的 key 名稱（如 yolo_pig_count_history），LLM 即可推論意圖。
  • 單位明示：溫度寫 "29.5°C" 而非裸數字，避免單位混淆。
  • 巢狀結構：將相關欄位群組（camera_analytics / environmental_sensors）幫助 LLM 分類。
  • 降噪：只放與報告相關的欄位，避免 prompt 過長浪費 token。

結構化數據就緒

本階段交付：

  • 每次觸發產生一份 JSON：內容含時序歷史與多維感測器讀數。
  • Token 友善：單次 JSON 通常 < 2k tokens，留出空間給 RAG 與 LLM 回應。
  • 可重現：同一時間區段重跑會得到相同 JSON，便於除錯與測試。
  • 進入下一階段：Agent 將同時拿到此 JSON 與 RAG 知識庫，開始推理。
