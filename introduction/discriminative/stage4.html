準備自訓練權重

從階段 2 取得的 best.pt 在桌機可運作，但直接部署到邊緣裝置會遇到瓶頸：

  • PyTorch 動態圖：推論時仍會經過 Python 解譯與動態 graph 建構，開銷大。
  • 嵌入式 GPU 算力有限：Jetson Nano 僅 472 GFLOPS（FP16），與桌機 RTX 等級差距百倍。
  • 記憶體頻寬：邊緣裝置記憶體與 GPU 共用，搬運成本高。
  • 解決方案：編譯為硬體優化的 TensorRT 引擎，跳過 Python 開銷並使用低精度運算。

CLI Export 編譯

Ultralytics 提供一行式指令完成模型轉換，主要參數：

  • model：來源 PyTorch 權重，可為相對或絕對路徑。
  • format=engine：輸出 TensorRT；其他選項：onnx、openvino、coreml、tflite。
  • device=0：編譯時使用的 GPU，建議在目標部署裝置上執行 export，因 TensorRT engine 與 GPU 架構綁定。
  • imgsz=640：輸入尺寸；engine 編譯後為固定尺寸，部署時須一致。
  • half=True：啟用 FP16 量化（詳見步驟 3）。

FP16 半精度量化

降低數值精度以換取效能，是邊緣部署的核心技巧：

  • FP32 → FP16：每個浮點數從 32 bit 縮減為 16 bit，記憶體佔用直接減半。
  • 硬體加速：NVIDIA Tensor Cores 對 FP16 有專屬加速通道，吞吐量約 2x。
  • 精度損失極小：對於物件偵測這類任務，mAP 通常僅下降 0.1~0.5%，肉眼無感。
  • 進階：INT8 量化：可再壓榨至 8 bit，但需提供 calibration dataset，且精度損失較顯著（適合更嚴苛場景）。

產出 TensorRT 引擎

best.engine 是針對特定 GPU 架構編譯的二進位檔：

  • kernel auto-tuning：編譯期間 TensorRT 會實際 benchmark 多種 kernel 實作，選最快版本。
  • layer fusion：合併連續層（如 Conv + BN + ReLU）為單一運算，減少 memory round-trip。
  • 不可跨架構：在 Jetson Orin 編譯的 engine 無法直接在 Jetson Nano 跑，需在每個目標裝置重新 export。
  • 使用方式：仍透過 YOLO('best.engine') 載入，介面與 .pt 完全相同，無需修改推論程式碼。

部署至邊緣裝置

常見的邊緣硬體與部署方式：

  • NVIDIA Jetson 系列：Orin Nano / Xavier NX 是工業界主流，原生支援 TensorRT。
  • 檔案傳輸：使用 scp best.engine drone@192.168.x.x:~/，或燒錄到 U 盤手動複製。
  • 自動啟動：撰寫 systemd unit 將推論腳本註冊為服務，開機後自動啟動並監聽攝影機。
  • 監控與日誌：透過 journalctl 或本地檔案保留執行紀錄，協助現場除錯。
  • 韌體更新：建議建立 OTA 機制，便於日後推送新版 engine。

完全離線即時推論

整套系統上線後的特性與優勢：

  • 零網路依賴：所有運算在無人機晶片完成，山區、地下、偏遠豬舍皆可運作。
  • 低延遲：單幀推論 ~8ms，加上影像 I/O 仍可維持 30fps 即時體驗。
  • 資料隱私：原始影像不離開裝置，符合農場資料保密需求。
  • 持續優化空間：蒐集現場誤判樣本回到 Stage 1 增量標註，形成資料閉環。
  • 專案完成 🎉：從採集到部署的完整 pipeline 至此交付，可開始實際營運。
